## 🎯 핵심 키워드

<aside>
💡 주요 내용들을 정리하고, 여러분만의 생각을 통해 정리해보세요!

- 레퍼런스를 참고하여 정의, 속성, 장단점 등을 적어주셔도 괜찮습니다.
- 조사는 공식 홈페이지 **Best**, 블로그(최신 날짜) **Not Bad**
</aside>

- OAuth 2.0
    
    OAuth 2.0은 사용자가 비밀번호를 직접 제공하지 않고도 외부 서비스(구글, 카카오 등)에 자신의 리소스 접근을 허용할 수 있게 하는 인가 프로토콜이다.
    구글 로그인처럼 타 서비스 계정으로 로그인하는 기능의 표준 방식이다.
    
- JWT (JSON Web Token)
    
    JWT는 서버가 상태를 저장하지 않고도 사용자 인증 정보를 안전하게 전달하기 위한 서명 기반 토큰이다.
    Header·Payload·Signature 구조로 되어 있으며 토큰만으로 인증이 가능해 확장성이 높다.
    
- Bearer Token
    
    Bearer Token은 HTTP 요청에서 “Authorization: Bearer ” 형태로 인증 정보를 전달하는 방식이다.
    토큰을 가진 사람을 신뢰하는 방식이므로 탈취되면 누구든지 사용할 수 있어 HTTPS가 필수적이다.

## ✨ 위클리 스크럼 과제

---

위클리 스크럼 과제는 OO주차 위클리 스크럼 아이스 브레이킹과 공통 질문을 모두 작성한 후, 아래에 질문을 복사 붙여넣기 한 다음 다같이 작성해주시면 됩니다. 

1. 8주차 미션으로 `handleUserSignUp` API에 `#swagger.requestBody`와 `#swagger.responses` 주석을 추가했습니다. 종종 **실제 코드보다 Swagger 주석이 더 길 때**도 있습니다. API 문서를 이렇게 '코드로' 관리하는 것이 귀찮게 느껴질 수도 있는데, 프론트엔드 개발자와의 **협업** 관점에서 Swagger 문서화가 왜 필수적이라고 생각하는지, Swagger UI에서 **가장 유용했던 기능**은 무엇인지 이야기해 봅시다.
2. 7주차에서는 `src/errors.js`에 커스텀 에러(`U001`)를 "정의"했고, 
    
    8주차에서는 `#swagger.responses[409]` 주석에 그 에러를 "문서화" 했습니다. 
    
    만약, 9주차에 새로운 `U002` (비밀번호 규칙 위반) 에러를 `src/errors.js`에 **추가**하고, Swagger 주석은 **수정하는 것을 잊어버린다면** 프론트엔드 개발자에게 어떤 문제가 발생할까요?

### 🍒 공통 미션

1. 기존에 사용자의 정보를 하드 코딩 했던 부분들이 있다면 수정해주세요! (ex. 항상 첫 번째 사용자를 가져와 이용했던 부분 등)
2. 기존 회원 가입 API의 경우, 이메일이 이미 존재하는 사용자는 회원 가입을 막아둔 상태입니다. 일단 Google 로그인 등으로 로그인 한 사용자의 경우 전화번호, 생일 등을 채울 마땅한 방법이 없는 상태입니다.
    - 기존 회원 가입 API를 수정하여 이미 존재하는 사용자도 정보를 채워 정보를 갱신할 수 있도록 수정해주셔도 괜찮고,
    - 새로운 API를 만들어서 자기 자신의 정보를 수정하는 API를 만들어주셔도 괜찮습니다. 자유롭게 선택해주세요!

### 🧙‍♂️ 시니어 미션

- 이번 주에는 Google 로그인 외에 다른 소셜 로그인을 하나 더 연동해주세요! 네이버 로그인, 카카오 로그인, Apple 로그인, GitHub 로그인 등 원하시는 로그인 방식을 연동해주시면 됩니다.
    - 어떠한 Passport 라이브러리를 선택했는지와, 연동해간 과정(스크린샷도 함께)을 미션 기록에 기록하며 진행해주세요!
- 이메일과 비밀번호를 이용한 회원가입, 로그인을 구현해주세요. `User` model에 `password` 관련 컬럼을 추가하고, https://www.passportjs.org/concepts/authentication/password/ - 문서를 참고하여 구현해주시면 됩니다. (비밀번호는 반드시 암호화되어 DB에 저장되어야 합니다.)
- 사용자가 최초로 회원가입 했을 때 사용한 회원가입 방식을 기록하고, 이 회원가입 방식 외의 다른 소셜 로그인 또는 이메일/비밀번호 로그인 및 비밀번호는 불가능하도록 방지해주세요. (Google 로그인으로 가입했다면 이후로도 Google로만 로그인이 가능하고, 이메일/패스워드로의 로그인은 불가능해야 합니다.)
- 워크북에서는 **JWT(Stateless)** 기반의 인증(Access/Refresh 흐름)을 배웠습니다. 이와 대비되는 **전통적인 인증 방식**으로는 'Cookie 기반의 **Session(Stateful)**'이 있습니다. 9주차 이론에서 간단히 짚고 넘어갔던 **'세션 기반 인증'**이 정확히 무엇인지, **JWT 방식과 비교했을 때 각각 어떤 장단점**이 있는지 찾아 정리해주세요.
